C++基本数据类型

  • 整数类型（Integer Types）：
    • int：整数类型，通常为32位。
    • short：短整数类型，通常为16位。
    • long：长整数类型，通常为32位（可能与int相同）。
    • long long：长长整数类型，通常为64位。
  • 浮点数类型（Floating-point Types）：
    • float：单精度浮点数。
    • double：双精度浮点数，通常比float精确度更高。
    • long double：扩展精度浮点数，精确度可能比double更高。
  • 字符类型（Character Types）：
    • char：字符类型，通常为一个字节。
    • wchar_t：宽字符类型，用于支持更广泛的字符集。
    • char16_t、char32_t：用于支持Unicode字符。
  • 布尔类型（Boolean Type）：
    • bool：布尔类型，只能取true或false两个值。
  • 空类型（Void Type）：
    • void：表示空类型，通常用作函数的返回类型或指针的占位符。

explicit关键字和强制类型转换

explicit使隐式申明不能调用。在C++中，explicit 是一种关键字，用于修饰类的构造函数。如果一个构造函数被 explicit 修饰了，就表示该构造函数只能以显式的方式进行调用，不能隐式地自动转换为其它类型，从而避免了隐式类型转换可能带来的问题。

使用 explicit 关键字修饰构造函数，可以确保该函数只能以显式的方式调用。这样可以避免隐式类型转换可能带来的歧义和错误，提高程序的可读性和安全性。

多态和继承

oop(Object-Oriented Programming)面向对象编程

多态性指的是在面向对象程序设计中，一个对象可以表现出多种类型和行为的能力。具体而言，它包括两个方面的含义：静态多态和动态多态。

静态多态通常通过函数重载和运算符重载来实现，同一函数或运算符可以接受不同的参数类型，从而实现不同的行为。

动态多态则是基于继承和虚函数的概念实现的。在这种情况下，派生类可以重写基类的虚函数，从而在调用时产生不同的行为。通过基类指针或引用来调用虚函数时，会根据指向的对象类型动态地选择要调用的函数版本，这样可以实现面向对象编程中的“接口规范化”和“多态替换”。

多态性的使用可以使程序更加灵活，代码的可重用性和可扩展性也得到了提升。

面向对象编程中的继承是指一个类（子类）可以从另一个类（父类或基类）继承其属性和方法。继承允许您创建一个新类，它可以从现有类中继承属性和行为，并且还可以添加自己的属性和行为。继承关系通常表示“is-a”关系，即子类是父类的一种类型。

在 C++ 中，使用关键字 class 或 struct 来定义类，派生类可以通过 class B : public A 的形式来继承基类 A。继承后，子类可以使用父类的公共成员函数和变量，也可以重写（override）父类中的成员函数，实现多态性。

有以下几种继承方式：

  1. 公有继承：派生类可以继承父类的公有成员，但不能继承私有成员。
  2. 私有继承：派生类可以继承父类的公有和保护成员，但都会变成私有成员。
  3. 保护继承：派生类可以继承父类的公有和保护成员，但都会变成保护成员。

继承是面向对象编程中重要的概念之一。它能够使代码结构更加清晰、易于理解和维护，同时也提高了代码的可重用性和可扩展性。

struct atomic，线程安全为什么这么重要

为了解决多线程编程中的数据竞争问题，C++11 引入了 std::atomic 类型，用于支持在多线程程序中对共享变量的原子访问，从而实现线程安全。

线程安全是指在多线程环境下，对共享资源的访问不会导致不正确的结果或线程间的数据竞争等问题。为了保证线程安全，可以采用以下几种方法：

  1. 互斥锁：使用互斥锁可以保证同一时间只有一个线程可以访问共享资源，其他线程需要等待锁的释放才能继续访问。互斥锁可以通过互斥量、信号量等方式实现。
  2. 原子操作：原子操作是指对共享变量的操作是不可分割的，可以保证操作的原子性，因此多个线程同时对共享变量进行操作不会导致数据竞争问题。在 C++11 标准中提供了 std::atomic 能够方便地进行原子操作。
  3. 读写锁：读写锁允许多个线程同时读取共享资源，但只允许一个线程进行写入访问。读写锁可以降低互斥锁带来的竞争开销，提高程序的并发性能。
  4. 线程局部存储（Thread Local Storage, TLS）：线程局部变量是指只在当前线程中有效的变量，在多线程环境下，每个线程都拥有自己独立的线程局部变量，避免了线程之间共享时的竞争问题。
  5. 无锁数据结构：无锁数据结构是指在无锁的情况下能够实现同步访问共享资源。无锁数据结构通常采用 CAS（Compare-And-Swap）等原子操作来确保读写操作的正确性，不需要显式地使用互斥锁或读写锁等同步机制。

以上方法都能够保证线程安全，但具体应该根据实际情况选择合适的方法。例如，在多读少写场景下可以使用读写锁提高程序性能，而在需要对共享资源进行复杂操作时可以考虑使用互斥锁来保证数据的正确性。

原子性是指一个操作或一组操作要么全部执行，要么全部不执行的特征。也就是说，原子性是指操作在执行过程中不会被中断，保证了操作的完整性和一致性。

互斥锁(Mutex Lock)互斥锁是一种最常见的线程同步机制，它可以保证同一时间只有一个线程可以访问共享资源，其他线程需要等待锁的释放才能继续访问。互斥锁通常使用互斥量（Mutex）来实现，通过加锁和解锁操作来控制对共享资源的访问。

读写锁(ReadWrite Lock)读写锁允许多个线程同时读取共享资源，但只允许一个线程进行写入访问。读写锁可以降低互斥锁带来的竞争开销，提高程序的并发性能。读写锁通常需要在加锁时指定是读操作还是写操作，读操作可以并发执行，写操作需要独占锁资源。

线程局部存储(Thread Local Storage,TLS)线程局部变量是指只在当前线程中有效的变量，在多线程环境下，每个线程都拥有自己独立的线程局部变量，避免了线程之间共享时的竞争问题。线程局部存储可以通过 C++11 新增的 thread_local 关键字来实现。

无锁数据结构(Lock-Free Data Structure)无锁数据结构是指在无锁的情况下能够实现同步访问共享资源。无锁数据结构通常采用 CAS（Compare-And-Swap）等原子操作来确保读写操作的正确性，不需要显式地使用互斥锁或读写锁等同步机制。无锁数据结构可以减少锁竞争问题，提高程序的并发性能。

智能指针

  1. shared_ptr

shared_ptr的实现机制是在拷贝构造时使用同一份引用计数,在所有指向同一地址的指针都被销毁后，才会释放对应空间。

重载了:operator*、operator->、copy constructor、operator=、~destructor。
是一个模板，必须new出来(次数+1)，指向的对象

shared_ptr包含两个部分:指向动态分配对象的指针和一个引用计数器。

引用计数器记录共享该对象的shared_ptr的数量。

安全问题:

(1) 同一个shared_ptr被多个线程“读”是安全的

(2) 同一个shared_ptr被多个线程“写”是不安全的

(3)共享引用计数的不同的shared_ptr被多个线程“写”是安全的

  1. unique_ptr

unique_ptr”唯一”拥有其所指对象。在同一时刻只能有一个unique_ptr指向给定对象，离开作用域时，若其指向对象，则将对象销毁。

不支持copy constructor、operator=。

定义时绑定到一个new返回的指针。

  1. weak_ptr

weak_ptr是为了配合shared_ptr而引入的一种智能指针。weak_ptr没有共享资源，它的构造不会引起指针引用计数的增加。

shared_ptr析构之后内存释放，在使用之前调用函数lock()检查weak_ptr是否为空指针。

shared_ptr缺陷和weak_ptr如何协同工作:

weak_ptr 是一种不持有对象所有权的智能指针，其构造函数接受一个shared_ptr对象，并通过成员函数lock获取这个shared_ptr对象

weak_ptr为弱引用类型,不会使引用计数增加，他的构造不增加引用计数，也不减少引用次数。被指引用就会+1 pa:1 pb:2

C++强制类型转换

  1. static_cast

静态转换可以用于基本类型、对象指针和引用之间的类型转换，可以将一个表达式从一个类型转换为另一种类型，并且在编译时检查类型是否安全。

  • 不能进行不安全或者非法类型的转换
  • 必须满足标注类型转换的所有条件
  • 上行转换(把派生类的指针或者引用转换成基类)是安全的,下行是不安全的(没有动态类型检查)
  1. dynamic_cast

在进行下行转换时,dynamic_cast具有类型检查(信息在虚函数中)的功能，比static_cast更加安全。动态转换可以用于类层次结构中的类型转换，它可以将派生类指针或引用转换成基类指针或引用，也可以将基类指针或引用转换成派生类指针或引用

  • 如果转换失败，返回的是一个空指针或空引用。引用时，转化失败抛出异常
  • 只能用于具有虚函数的类，因为静态类型和动态类型不一致时，只有具有虚函数的类才能根据虚表中的信息来进行类型转换。
  • 转换后必须是类的指针、引用或者void*
  1. reinterpret_cast

重新解释转换可以将任何指针转换为任何其他指针类型，通常用于将指针或整数类型相互转换。

  • 可能导致运行时错误，因为它强制修改了数据的类型，不经过检查。
  • 可以将一个指针类型转换为一个整数类型。
  • 不应该将整数类型转换为指针类型，因为这可能会导致未定义的行为。
  1. const_cast

常量转换可以用于删除表达式中的 const 或 volatile 属性，也可以在指针或引用之间转换。

  • 不能用于修改有 const 属性的数据。
  • 可以用于去掉 const 属性并修改非 const 数据。
  • 可以将指向非 const 的指针或引用转换成指向 const 的指针或引用，并且不会失去数据的 const 属性。

内存泄漏

内存泄漏(memory leak)指由于疏忽或错误造成了程序未能释放掉不再使用的内存。失去了对该段内存的控制，造成了内存的浪费。

使用Valgrind,mtrace进行内存泄漏检查。

内存泄漏分类:

  • 堆内存泄漏

程序运行中根据需要使用malloc、realloc new等从堆中分配一块内存，再完成后调用free、delete释放。如果没有被释放，那么该块内存将不会被使用，产生Heap Leak。

  • 系统资源泄漏

Resource Leak，主要指程序使用系统分配的资源，比如Bitmap,handle,SOCKET等没有使用相应的函数释放掉，导致系统资源的浪费，严重可导致系统效能降低，系统运行不稳定

  • 没有将基类的析构函数定义为虚函数

当基类指针指向子类对象时，如果基类的析构函数不是virtual，那么子类的析构函数将不会被调用，子类的资源没有正确释放，因此造成内存泄漏。

防止内存泄漏的一些方法:

  • 将内存的分配服装在类中
  • 构造函数分配内存
  • 析构函数释放内存
  • 使用智能指针

计算机中的乱序执行

因为编译器会对代码进行优化，所有可能会存在很多细小的差别或错误，例如volatile，所有需要知道计算机一定的优化方式，以编写健壮性的代码。乱序执行是指计算机程序中，代码指令的执行顺序和程序设计者原先所写的顺序不同。

1.一定会按正常顺序执行的情况

  • 对同一块内存进行访问，此时访问的顺序不会被编译器修改
  • 新定义的变量依赖于之前定义的变量，此时两个变量定义的顺序不会被编译器修改

2.其他情况计算机会进行乱序执行

单线程的情况下允许，但是多线程情况下就会产生问题

c++中的库提供了六种内存模式用于多线程的情况下防止编译器的乱序执行

(1)memory_order_relaxed

(2)memory_order_consume

(3)memory_order_acquire

(4)memory_order_release

(5)memory_order_acq_rel(acquire/release)

(6)memory_order_seq_cst(squentially consistent)

好处：不需要编译器设置内存屏障，morden c++开始就会有底层汇编能力

线程安全

线程安全是指在多线程环境下，对共享资源的访问不会导致不正确的结果或线程间的数据竞争等问题。为了保证线程安全，可以采用以下几种方法：

  1. 互斥锁：使用互斥锁可以保证同一时间只有一个线程可以访问共享资源，其他线程需要等待锁的释放才能继续访问。互斥锁可以通过互斥量、信号量等方式实现。
  2. 原子操作：原子操作是指对共享变量的操作是不可分割的，可以保证操作的原子性，因此多个线程同时对共享变量进行操作不会导致数据竞争问题。在 C++11 标准中提供了 std::atomic 能够方便地进行原子操作。
  3. 读写锁：读写锁允许多个线程同时读取共享资源，但只允许一个线程进行写入访问。读写锁可以降低互斥锁带来的竞争开销，提高程序的并发性能。
  4. 线程局部存储（Thread Local Storage, TLS）：线程局部变量是指只在当前线程中有效的变量，在多线程环境下，每个线程都拥有自己独立的线程局部变量，避免了线程之间共享时的竞争问题。
  5. 无锁数据结构：无锁数据结构是指在无锁的情况下能够实现同步访问共享资源。无锁数据结构通常采用 CAS（Compare-And-Swap）等原子操作来确保读写操作的正确性，不需要显式地使用互斥锁或读写锁等同步机制。

以上方法都能够保证线程安全，但具体应该根据实际情况选择合适的方法。例如，在多读少写场景下可以使用读写锁提高程序性能，而在需要对共享资源进行复杂操作时可以考虑使用互斥锁来保证数据的正确性。

volatile和std::atomic

volatile主要使用场景

  1. 多线程编程

在多线程编程中，如果一个变量被多个线程访问，那么这个变量就可能出现同步问题。为了解决这个问题，可以使用 volatile 来确保变量的可见性，防止编译器将该变量缓存到 CPU 的寄存器或高速缓存中，从而使得每个线程读取的都是最新的值，而不是缓存的旧值。

  1. 中断处理程序

在嵌入式系统中，硬件设备常常会发生中断，此时 CPU 必须及时响应中断服务程序，获取设备状态等信息并做出相应的处理。在中断处理程序中，由于涉及到硬件操作，因此需要使用 volatile 来确保变量的可见性和正确性。

atomic主要使用场景

std::atomic 是 C++11 中提供的一个原子类型库，它的主要作用是在多线程程序中保证数据的原子性，防止多个线程同时访问和修改同一份数据时出现竞态条件（Race Condition）而导致的数据不一致问题。简单来说，std::atomic 可以确保多线程程序中的共享变量能够正确地被读写，避免了由于多线程访问导致的数据竞争问题。如:内存池、计数器、状态机、标志位。

区别：

虽然 volatile 和 std::atomic 都可以用于确保多线程编程中变量的可见性和正确性，但是它们之间还有一些细微的差别。

  1. 并发写入

在并发写入的场景下，volatile 不能保证原子性。当多个线程同时写入一个 volatile 变量时，并发写入可能会导致数据丢失或出现操作冲突，从而导致数据的不一致性。而 std::atomic 则提供了一些原子操作接口，如 store() 和 exchange() 方法等，可以保证多个线程同时写入时操作的原子性，从而避免了并发写入时可能出现的问题。

  1. 操作类型

volatile 仅能用于修饰基本数据类型和指针，而不能用于自定义类型。而 std::atomic 则提供了一个模板类，可以用于定义任意类型的原子变量，包括自定义类型。

  1. 竞态条件

在存在竞态条件的情况下，volatile 不能保证数据的正确性。例如，在 ++ 操作中，多个线程同时读取变量、计算新值并写入变量，这样可能会导致数据不一致的问题。而 std::atomic 提供了 fetch_add()，compare_exchange_strong() 等方法，可以保证这种竞态条件下的操作的原子性和正确性。

  1. 性能

由于 volatile 仅保证变量的可见性和禁止编译器优化，而没有提供原子操作接口，因此在执行复杂的并发操作时，其效率可能不如 std:atomic。

综上所述，虽然 volatile 和 std::atomic 都可以用于确保多线程环境下变量的可见性和正确性，但是它们之间还是存在一些微小的差别，应该根据具体情况选择合适的方式来确保程序的正确性和高效性。

三次握手（three-way handshake)

TCP建立一个连接需要3个分节

(1)服务器必须准备好接受外来的连接。这通常通过调用socket、bind、listen这3个函数来完成，我们称其为被动打开(passive open)

(2)客户通过调用connect发起主动打开。这导致客户TCP发送一个SYN（同步）分节，它告诉服务器客户将在（待建立的)连接中发送的数据的初始序列号。通常SYN分节不携带数据，其所在IP数据报只包含了一个IP首部，一个TCP首部及可能有的TCP选项。

(3)服务器必须确认(ACK)客户的SYN，同时自己也得发送一个SYN分节。它含有服务器将在同一连接中发送的数据的初始序列号。

(4)客户必须确认服务器的SYN。

四次挥手

(1)某个应用进程首先调用close，我们称该端执行主动关闭(active close)。该端的TCP于是发送一个FIN分节，表示数据发送完毕。

(2)接收到这个FIN的对端咨询被迫关闭(passive close)。这个FIN由TCP确认。它的接收也作为一个文件结束符(end-of-file)传递给接收端应用进程（放在已排队等待该应用进程接收的任何其他数据之后），因为FIN的接收意味这接受应用进程在相应连接上再无额外数据可接受。

(3)一段时间后，接收到这个文件结束符的应用进程将调用close关闭它的套接字。这导致它的TCP也发送一个FIN。

(4)接收这个最终FIN的原发送端TCP(即执行主动关闭的那一端)确认这个FIN。

段错误

段错误(Segmenttation Fault,缩写为SegFault)，也称为访问违例(Access Violation),指程序视图访问未分配给它的内存位置，或试图向只读的内存位置写入数据，从而导致操作系统发出信号终止程序运行的一种错误。

常见情况下，当程序试图访问不存在的内存地址，或试图在受到保护的内存区域中进行修改操作时，操作系统会将其识别为非法操作，并向该进程发送一个信号，以示程序运行异常。如果该进程没有处理这个信号，那么操作系统会强制终止该进程，并生成一个核心转储文件（core dump file）以供调试。

发生段错误通常是由于程序中存在以下问题之一：

  1. 访问了已经被释放的内存空间；
  2. 访问了未被初始化的指针变量；
  3. 数组越界访问；
  4. 访问了只读的内存空间；
  5. 栈溢出等。

为了避免发生段错误，我们应该编写健壮的代码，合理地使用指针、数组和内存管理等相关知识，并进行严格的错误检查和异常处理。

模板特化(Template Specialization)

模板特化(Template Specialization)是一种机制,它允许你根据特定类型或特点条件提供定制化的模板实现。通过模板特化，你可以为某个特定的类型或条件提供一个专门的模板实现，使其在特定情况下得到特殊处理。

1.部分特化(Partial Specialization):

你需要为某个模板类或模板函数提供多个不同的实现时，可以使用部分特化。这种特化方式通常基于类型的某些特性或条件进行选择，以提供更具体的实现。

例如，对模板类进行部分特化可以根据类型特征（如指针、引用等）或条件（如大小、属性等）来选择不同的实现。

2.显式特化(Explicit Specialization):

当你需要为特定类型提供唯一的模板实现时，可以使用显式特化。显式特化是对整个模板进行特化，并提供一个非通用的实现。

例如，你可以为特定类型提供一个特殊的模板实现，以覆盖默认的通用实现或为其提供更高效的版本。

需要注意的是，模板特化应该与通用模板进行区分，并且在正确的上下文中使用。此外，模板特化的实现应该根据具体的需求和条件进行设计，以获得预期的特殊行为或优化效果。

编译为可执行程序的过程

1.预处理(Preprocessing):在编译之前，预处理器会对源代码进行处理。它会根据以#开头的预处理指令，如包含头文件（#include）、宏定义（#define）、条件编译（#ifdef、#ifndef、#if等）等来对源代码进行替换、扩展和删除等操作。

2.编译(Compilation):经过预处理后，编译器将源代码翻译为汇编语言或者直接翻译为机器语言的中间表示，并生成相应的目标文件。编译器会进行语法分析、语义分析、类型检查、代码优化等操作，将高级语言的代码转换为低级语言的表示形式。

3.汇编(Assembling):汇编器将目标文件中的汇编语言代码转换成机器指令，并生成相应的目标文件。这些机器指令是由特定的 CPU 架构所能理解和执行的。

4.链接(Linking):如果程序中有多个源文件或者使用了外部的库文件，链接器将把编译得到的多个目标文件以及可能需要的库文件合并成一个单独的可执行文件。在链接的过程中，符号引用和定义会被解析，重复的代码和数据会被消除，最终生成可执行文件。

5.加载(Loading):将可执行文件加载到内存中，并为程序分配所需的资源。加载器负责将可执行文件中的指令和数据加载到适当的内存位置，并设置程序的执行环境。

6.执行(Execution):操作系统启动程序的执行，根据程序入口点开始执行指令，程序按照指令顺序执行，直到程序结束或者遇到特定的终止条件。

进程间通信

进程间通信（Inter-Process Communication，IPC）是指不同进程之间进行数据交换和通信的机制。在操作系统中，进程是独立运行的程序实例，它们拥有各自的内存空间和执行环境。为了实现不同进程之间的协作和数据共享，需要使用IPC机制。

常见的进程间通信方法包括：

  1. 管道（Pipe）：管道是一种最基本的IPC机制，它提供了一个半双工的通信通道。在父进程中创建管道后，可以通过fork创建子进程，并通过文件描述符进行读写操作，实现进程间的通信。
  2. 命名管道（Named Pipe）：命名管道也是一种基于管道的通信机制，但允许无亲缘关系的进程之间进行通信。它通过一个特殊的文件来实现进程间的数据交换。
  3. 消息队列（Message Queue）：消息队列是一种通过消息传递进行进程间通信的机制。进程可以通过消息队列发送和接收具有特定格式的消息，实现异步通信和数据共享。
  4. 共享内存（Shared Memory）：共享内存是一种高效的IPC方式，它允许多个进程共享同一块物理内存区域。进程可以直接读写共享内存区域，无需通过数据拷贝或中间缓冲区。
  5. 信号量（Semaphore）：信号量是一种计数器，用于实现多个进程之间的同步和互斥操作。进程可以通过信号量进行临界区的保护，避免竞态条件和资源争用。
  6. 套接字（Socket）：套接字是一种网络编程中常用的IPC机制，它提供了进程间通过网络进行通信的能力。套接字支持在不同主机之间进行进程间通信。

这些IPC机制各有优缺点，选择合适的IPC方法取决于具体的应用场景和需求。需要根据进程之间的关系、通信模式、性能要求等因素来进行选择和设计。

UML类之间的关系

在UML（统一建模语言）中，类之间的关系可以通过不同的箭头和符号来表示。以下是常见的类之间关系及其表示方法：

  1. 关联关系（Association）：用实线箭头表示。表示一个类与另一个类有关联，通常包括单向关联和双向关联。强依赖关系。比如我和我的朋友。关系比较平等的，长期性的。
  2. 聚合关系（Aggregation）：用空心菱形加实线箭头表示。表示一个类包含另一个类，但是另一个类可以存在独立于当前类的实例。has-a(当然可以扔掉哦)
  3. 组合关系（Composition）：用实心菱形加实线箭头表示。表示一个类包含另一个类，且另一个类的实例的生命周期与当前类的实例相依赖。代码上一致，只能去语义区分
  4. 继承关系（Inheritance）：用空心三角形加实线箭头表示。表示一个类继承自另一个类，即子类继承父类的属性和方法。
  5. 实现关系（Realization）：用虚线箭头表示。表示一个类实现了一个接口或者抽象类定义的方法。
  6. 依赖关系（Dependency）：用虚线箭头表示。表示一个类依赖于另一个类，通常是一个类作为另一个类的参数、返回值或局部变量使用。不能独立出来
  7. 限制关系（Constraint）：用大括号{}表示。表示对类之间关系的进一步约束或条件，常在箭头旁边附加限制说明。

is-a(继承关系):表示一个类是另一个类的特殊类型或子类

has-a(聚合关系):表示一个类具有其他类的对象作为其属性或成员变量。一个类的实例包含了另一个类的实例，形成整体和部分的关系。

use-a(使用关系):表示一个类使用另一个类的功能，但不具有继承或组合关系。一个类通过调用另一个类的方法和访问其成员来使用它。

“contains-a”（组合关系）：一个类包含另一个类，且另一个类的实例的生命周期与当前类的实例相依赖。例如，人和人的大脑。代码上一致，只能去语义区分

linux是4种锁机制

**互斥锁:**mutex,用于保证在任何时刻，都只有一个线程访问该对象。当获取锁操作失败时，线程会进入睡眠，等待锁释放时被唤醒。

读写锁:rwlock,分为读锁和写锁。处于读操作时，允许多个线程同时获得读操作。但同以时刻只能有一个线程可以获得写锁。其他获取写锁失败的线程会进入睡眠状态，直到写锁释放时被唤醒。写锁会阻塞其他读写锁。当有一个线程获得写锁在写时，读锁也不能被其他线程获取。写者优先读者。

**自旋锁:**spinlock，在任何时刻同样只能有一个线程访问对象。但是当获取锁操作失败时，不会进入睡眠，而是会在原地自旋，直到锁被释放。这样节省了线程从睡眠状态到被唤醒期间的消耗，在加锁时间短暂的环境下会极大的提高效率。但如果加锁时间过长，则会非常浪费CPU资源。

**RCU:**即read-copy-update，在修改数据时，首先需要读取数据，然后生成一个副本，对副本进行修改。修改完成后，再将老数据update成新的数据。使用RCU时，读者几乎不需要同步开销，既不需要获得锁，也不使用原子指令，不会导致锁竞争，因此就不用考虑死锁问题了。而对于写者的同步开销较大，它需要复制被修改的数据，还必须使用锁机制同步并行其它写者的修改操作。在有大量读操作，少量写操作的情况下效率非常高。

指针常量和常量指针

常量指针(pointer to a constant,底层const):定义了一个指针，这个指针指向一个只读的对象，不能通过常量指针来改变这个对象的值。常量指针强调的是指针对其所指对象的不可改变性。星号靠近变量名，const在*前面,指针不能修改地址的内容

const int* temp或者int const* temp

指针常量(constant pointer,顶层const):定义了一个指针，这个指针的指只能在定义时初始化，其他地方不能改变。指针常量强调的是指针的不可改变性。靠近变量类型。指针固定了指向的地址

int* const temp = &a;

堆和栈的区别

  • 申请方式不同

    • 栈是由系统自动分配
    • 堆是自己申请和释放

  • 申请大小限制不同

    • 栈顶和栈底是之前预设好的，栈是向栈底扩展，大小固定
    • 堆向高地址扩展，是不连续的内存区域，大小可以灵活调整

  • 申请效率不同

    • 栈由系统分配，速度快，不会有碎片
    • 堆由程序员分配，速度慢，且会由碎片

指针和引用的区别

  • 指针是一个变量，存储的是一个地址，引用跟原来的变量实质上是同一个东西，是原变量的别名
  • 指针可以多级，引用只有一级
  • 指针可以为空，引用不能为空且在定义时必须初始化
  • 指针在初始化后可以改变指向，而引用在初始化之后不可再改变
  • sizeof指针得到的是本指针的大小，sizeof引用得到的是引用所指向变量的大小
  • 当把指针作为参数进行传递时，也是将实参的一个拷贝传递给形参，两者指向的地址相同，但不是同一个变量，在函数中改变这个变量的指向不影响实参，而引用却可以。
  • 引用本质是一个指针，同样会占4字节内存；指针是具体变量，需要占用存储空间（，具体情况还要具体分析）。
  • 引用在声明时必须初始化为另一变量，一旦出现必须为typename refname &varname形式；指针声明和定义可以分开，可以先只声明指针变量而不初始化，等用到时再指向具体变量。
  • 引用一旦初始化之后就不可以再改变（变量可以被引用为多次，但引用只能作为一个变量引用）；指针变量可以重新指向别的变量。
  • 不存在指向空值的引用，必须有具体实体；但是存在指向空值的指针

什么使用指针和引用:

  • 需要返回函数内局部变量的内存的时候用指针。使用指针传参需要开辟内存，用完要记得释放指针，不然会内存泄漏。而返回局部变量的引用是没有意义的
  • 对栈空间大小比较敏感（比如递归）的时候使用引用。使用引用传递不需要创建临时变量，开销要更小
  • 类对象作为参数传递的时候使用引用，这是C++类对象传递的标准方式

数据库中的乐观锁和悲观锁

  • 一般的数据库都回支持并发操作，在并发操作中为了避免数据冲突，所以需要对数据上锁，乐观锁和悲观锁就是两种不同的上锁方式
  • 悲观锁假设数据在并发操作中一定会发生冲突，所以在数据开始读取的时候就把数据锁住。而乐观锁则假设数据一般情况下不会发生冲突，所以在数据提交更新的时候，才会检测数据是否有冲突。
  • 悲观锁的实现：悲观锁有行级锁和页级锁两种形式。行级锁对正在使用的单条数据进行锁定，事务完成后释放该行数据，而页级锁则对整张表进行锁定，事务正在对该表进行访问的时候不允许其他事务并行访问。
    悲观锁要求在整个过程中一直与数据库有一条连接，因为上一个事务完成后才能让下一个事务执行，这个过程是串行的。
  • 乐观锁的三种常用的实现形式:
    • 一种是在执行事务时把整个数据都拷贝到应用中，在数据更新提交的时候比较数据库中的数据与新数据，如果两个数据一摸一样则表示没有冲突可以直接提交，如果有冲突就要交给业务逻辑去解决。
    • 一种是使用版本戳来对数据进行标记，数据每发生一次修改，版本号就增加1。某条数据在提交的时候，如果数据库中的版本号与自己的一致，就说明数据没有发生修改，否则就认为是过期数据需要处理。
    • 最后一种采用时间戳对数据最后修改的时间进行标记。

程序编译为可执行文件的过程

预处理，编译，汇编，链接

1.预处理(Preprocessing):对预处理命令进行替换等预处理操作

生成*.i文件(intermediate,中间)。对#开头的预编译指令进行处理

2.编译(Compilation):代码优化和生成汇编代码

生成.s文件。对编译生成后的.i/.ii文件进行词法分析、语法分析、语义分析及优化

3.汇编(Assembly):将汇编代码转化为机器语言

生成.o/.so文件。根据汇编指令和机器指令的对照表，进行翻译。

4.链接(Linking):将目标文件彼此链接起来

生成.exe或没拓展名。

将不同的源文件产生的目标文件进行链接，从而形成一个可以执行的程序。可分为动态和静态链接。

  • 静态链接:函数和数据被编译进一个二进制文件。在使用静态库的情况下，在编译链接可执行文件时，链接器从库中复制这些函数和数据并把它们和应用程序的其它模块组合起来创建最终的可执行文件。
  • 动态链接:把程序按照模块拆分成各个相对独立部分，在程序运行时才将它们链接在一起形成一个完整的程序，而不是像静态链接一样把所有程序模块都链接成一个单独的可执行文件。

网络七层协议/OSI七层模型

物理层、数据链路层、网络层、传输层、会话层、表示层、应用层。

物理层(Physical Layer):解决两个硬件之间如何通信的问题，常见的物理媒介有光纤、电缆、中继器等。主要定义物理设备标准，如网线的接口类型、传输速率等。负责传输原始比特流(bit stream)

数据链路层(Data Link Layer):通知各种控制协议，将有差错的物理信道变为无差错的，能可靠传输数据帧(date frame)的数据链路。处理单位是帧，通过物理地址(MAC地址)进行寻址和错误检查

网络层(Network Layer):通过路由选择算法，为报文(该层的数据单位，由上一层数据打包而来)通过通信子网选择最适当的路径。定义的是IP地址，通过IP地址寻址，所以产生了IP协议。负责将数据包(packet)从源节点传输道目标节点，通过路由选择算法进行路径选择和转发。使用IP地址寻找，实现了跨网络通信

传输层(Transport Layer):监控数据传输服务的质量，保证报文的正确传输。提供端到端的可靠数据传输服务。将数据分割成小的数据段，并使用端口号对应用程序进行标识。主要协议由TCP和UDP

会话层(Session Layer):建立和管理应用程序之间的通信。虽然已经可以实现给正确的计算机，发送正确的封装过后的信息了。但我们总不可能每次都要调用传输层协议去打包，然后再调用IP协议去找路由，所以我们要建立一个自动收发包，自动寻址的功能。负责处理会话的建立和维护包括会话的开始、同步、暂停和恢复等功能。

表示层(Presentation Layer):负责数据格式的转换，将应用处理的信息转换为适合网络传输的格式，或者将来自下一层的数据转换为上层能处理的格式。负责数据的格式化、加密和压缩等操作，以确保应用程序能够理解传输的数据。提供了数据的语法和语义的独立性

应用层(Application Layer):是计算机用户，以及各种应用程序和网络之间的接口。功能是直接向用户提供服务，完成用户希望在网络上完成的各种工作。直接与用户的应用程序进行交互

OSI七层模型:物理层、数据链路层、网络层、传输层、会话层、表示层、应用层

TCP/IP五层模型：物理层、数据链路层、网络层、传输层、应用层

TCP/IP四层模型:网络接口层、网络层、传输层、应用层

DNS是什么

DNS(Domain Name System,域名系统)，因特网上作为域名和IP地址相互映射的一个分布式数据库，能够使用户更方便的访问互联网，而不用去记住能够被机器直接读取的IP数串。通过主机名，最终得到该主机名对应的IP地址的过程叫做域名解析（或主机名解析）。通俗的讲，我们更习惯于记住一个网站的名字，比如www.baidu.com,而不是记住它的ip地址，比如：167.23.10.2。

HTTP请求方式

HTTP1.0：GET、POST、HEAD

HTTP1.1：OPTIONS、PUT、PATCH、DELETE、TRACE、CONNECT

GET 请求指定的页面信息，并返回实体主体
HEAD 类似于GET，只不过返回的响应中没有具体的内容，用语获取报头
POST 向指定资源提交数据进行处理请求(如提交表单或者上次文件)。数据被包含在请求体中。POST请求可能会导致新的资源的建立和/或已有资源的修改
PUT 从客户端向服务器传送的数据取代指定的文档的内容
DELETE 请求服务器删除指定的页面
CONNECT HTTP/1.1协议中预留给能够将连接改为管道方式的代理服务器
OPTIONS 允许客户端查看服务器的性能
TRACE 回显服务器收到的请求，主要用于测试或诊断
PATCH 对PUT方法的补充，用来对已知资源进行局部更新

struct和class的区别

相同:

  • 两者都拥有成员函数、公有和私有部分
  • 如何可以使用class完成的工作，struct也可

不同:

  • 两者中如果不对成员不指定公私有，struct默认是公有的，class则默认私有的
  • class默认是private继承，struct默认是public继承

C++和C的struct区别

  • C语言中：struct是用户自定义数据类型（UDT）；C++中struct是抽象数据类型（ADT），支持成员函数的定义，（C++中的struct能继承，能实现多态）
  • C中struct是没有权限的设置的，且struct中只能是一些变量的集合体，可以封装数据却不可以隐藏数据，而且成员不可以是函数
  • C++中，struct增加了访问权限，且可以和类一样有成员函数，成员默认访问说明符为public（为了与C兼容）
  • struct作为类的一种特例是用来自定义数据结构的。一个结构标记声明后，在C中必须在结构标记前加上struct，才能做结构类型名（除：typedef struct class{};）;C++中结构体标记（结构体名）可以直接作为结构体类型名使用，此外结构体struct在C++中被当作类的一种特例

成员初始化列表会在什么时候用到？它的调用过程是什么?

  1. 当初始化一个引用成员变量时
  2. 初始化一个const成员变量时
  3. 当调用一个基类的构造函数，而构造函数拥有一组参数时
  4. 当调用一个成员类的构造函数，而他拥有一组参数时

Effective C++

为什么学习c++,c++难学易用，拥有广博的语法，能够实现各种各样的东西,巨大而变化多端的设计可以表现出来，同时也有可能导致你的代码难以理解、不易维护、不易扩充等。本文章基于《Effective c++》，《Effective Modern c++》等各类数据及网络文章学习总结而得。

1.将c++视为语言联邦，而非单一语言

c++一开始的名称为c with classes，随着后续的更新，越来越成熟，引入了各种各样的特性，c++有4个主要部分

  • C
  • Object-Oriented C++
  • Template C++
  • STL

c++以c为基础，拓展了许多的功能。Object-Oriented c++为这门语言带来了class、encapsulation（封装）、inheritance(继承)、polymorphism(多态)、virtual(动态绑定)。Template C++为这门语言带来了generic programming(范型编程)，为代码的重用提供了基础。STL是个template程序库，对contaimers(容器)、iterators(迭代器)、algorithms(算法)、function objects(函数对象)的规约有极佳的紧密配合与协调。

13.以对象管理资源

Use objects to manage resources.

C++程序中最常使用的资源就是动态分配内存。其他常见资源：文件描述器(file descriptors)、互斥锁(mutex locks)、数据库的连接、图像资源、网络sockets。在不再使用后，都应该将它还给系统。

RAII(Resource Acquisition Is Initialization)资源取得时机便是初始化时机。

将资源管理写入类中，让类调用析构函数来管理资源的释放，类似auto_ptr,但auto_ptr已经被弃用。 auto_ptr已经弃用，且用auto_ptr初始化会为NULL

RCSP(reference-counting smart pointer)引用计数型智能指针。shared_ptr。

当然，手动删除可能出现问题

  • 为防止资源泄漏，请使用RAII对象，它们在构造函数中获得资源，在析构函数中释放资源。
  • 两个常被使用的RAII classes分别是shared_ptr和auto_ptr。前者直观，后者弃用，且复制会变为NULL。

14.在资源管理类中小心copying行为

Think carefully about copying behavior in resource-manageing classes.

但不想使用默认生成的拷贝构造函数，那么就得自己生成为private。

  • 禁止复制(copying声明为private)
  • 对底层资源使用“引用计数法”（reference-count）
  • 复制底部资源(浅拷贝)
  • 转移底部资源的拥有权(移动构造)

copying函数（包含copy构造函数和copy assignment操作符）有可能被编译器自动创建出来，因此除非编译器所生版本做了你想要做的事，否则得我们自己编写。智能指针中有原始指针的复件

  • 复制RAII对象必须一并复制它所管理的资源，所有资源的copying行为决定RAII对象的copying行为
  • 普遍而常见的RAII class copying行为是：抑制copying、施行引用计数法(reference counting)。不过其他行为也都可能被实现。

15.在资源管理类中提供原始资源的访问

Provide access to raw resources in resource-managing classes.

管理类存放的是资源的指针，我们无法从管理类直接得到一个资源对象（只能得到一个指针，通过指针找到对象）我们也不应该获得资源本身。所以我们最好用显式转化或者隐式转换（自动类型转换）来得到一个资源对象

  • APIs往往要求访问原始资源（raw resources)，所以每一个RAII class应该提供一个“取得其所管理之资源”的办法
  • 对原始资源的访问可能经由显式或隐式转换。一般而言显式更安全，隐式更方便。

16.成对使用new和delete时要采用相同形式

Use the same form in corresponding uses of new and delete.

使用new动态生成一个对象时，有两件事情发生

  • 分配内存(operator new)
  • 调用对应的构造函数

使用delete释放一个对象时，也有两件事情发生

  • 调用对应的析构函数
  • 释放内存(operator delete)

对于数组和对象都应该采用对应的释放方式来释放。错误的释放方式是有害的

故对数组尽量不要使用typedef/using方式。

Prefer pass-by-reference-to-const to pass-by-value.

pass-by-value的缺陷：

1.增加了构造和析构成本

2.切割(slicing)问题

参数w会构造一个Window对象，pass-by-value，WWSB的特化信息会被切除。因此，调用的总是Window::display:disappointed:。所有使用pass-by-reference-to-const。

当然也有使用pass-by-value，内置类型使用value的效率更高，编译器不同对内置类型和用户自定义类型的态度也有可能不同。

  • 尽量使用pass-by-reference-to-const替换pass-by-value
  • 注意内置类型，以及STL的迭代器和函数对象。

21.必须返回对象时，别妄想返回其reference

Don’t try to return a reference when you must return an object.

22.将成员变量声明为private

Declare data members private.

将成员变量升级为properity。隐藏了内部成员的实现细节，通过访问函数提供了一种统一的方式来读取和设置属性值。

1.对成员变量的控制更加精准

2.类具有封装性

属性的优点：

  • 封装性(Encapsulation)：属性可以隐藏成员变量的实现细节，只提供访问函数接口。这使得类能够更好地控制属性的使用和修改。
  • 可扩展性(Extensibility)：属性可以在后期扩展或修改其访问函数实现，而无需更改使用该属性的代码。这提供了一种更灵活的方式来管理类的数据。

23.宁可以non-member、non-friend替换member函数

Prefer non-member non-friend functions to member

假设有一个class表示网页浏览器

以上两种哪一个比较好了？

面向对象守则要求，数据以及操作数据的函数应该捆版在一起，意味着member函数是较好的选择。但member比non-member的封装性低。non-member允许有较大的包裹弹性(packaging flexibility)，而最终导致较低的编译相依度，增加webBrowser的可延伸性。member增加了对对象内的数据访问，数据的封装性就越低。func相当于一个便利函数。

  • 宁可拿non-member non-friend函数替换member函数。这样能增加封装性、包裹弹性和机能扩充性

优化版本：声明一个non-member swap让它调用member swap,但不再将那个non-member swap声明为std::swap的特化版本或重载版本。

当swap缺省版的效率不足：

  • 提供一个public swap函数，提高效率，且该函数不该抛出异常；
  • 在该命名空间中提供一个non-member swap函数，且调用swap成员函数；
  • 若调用swap，则记得包含using声明式；

建议：

  • 当std::swap对你的类型效率不高时，需提供一个swap成员函数，且该函数不能抛出异常
  • 若你提供一个member swap，也该提供一个non-member swap来调用member swap,对于classes，需特化std::using;
  • 当std::swap对你的类型效率不高时，提供一个swap成员函数，并确定这个函数不抛出异常。
  • 如果你提供一个member swap,也该提供一个non-member swap用来调用前者。对于classes(而非templates).也请特化std::swap
  • 调用swap时应针对std::swap使用using声明式，然后调用swap并且不带任何”命名空间资格修饰”
  • 为“用户定义类型”进行std templates全特化是好的，但千万不要尝试在std内加入某些对std而言全新的东西

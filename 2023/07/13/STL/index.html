STL

C++ STL

STL(Standard Template Library):标准模板库

标准库以header files形式呈现：

  • C++标准库的header files不带副档名(.h):#include < vector>
  • 新式C header files 不带副档名.h:#include
  • 旧式C header files 带有副档名.h,仍可用:#incldue < stdio.h >
  • 新式headers内的组件封装于namespace std
    • using namespace std::
    • using std::cout
  • 旧式headers内的组件不封装于namespace std;

STL六大组件:

  • 容器(Containers)
  • 算法(Algorithms)
  • 迭代器(Iterators)
  • 仿函数(Functors)
  • 适配器（配接器)(Adapters)
  • 空间配置器(Allocators)

STL六大组件的交互关系:

1.容器通过空间

2.算法通过迭代器存储容器中的内容

3.仿函数可以协助算法完成不同的策略的变化

4.适配器可以修饰仿函数

容器：各种数据结构，如vector、list、deque、set、map等，用来存放数据，从实现角度来看，STL容器是一种class template。(数据)

算法：各种常用的算法，如sort、find、copy、for_each。从实现的角度来看，STL算法是一种function tempalte。(操作)

迭代器：扮演了容器与算法之间的胶合剂，共有五种类型，从实现角度来看，迭代器是一种将operator*,operator->,operator++,operator–等指针相关操作予以重载的class template。(泛化的指针)

仿函数：行为类似函数，可作为算法的某种策略。从实现角度来看，仿函数是一种重载了operator()的class或者class template。(作用像个函数)

适配器：一种用来修饰容器或者仿函数或迭代器接口的东西。STL提供的queue和stack,虽然看似容器，但其实只能算是一种容器适配器，因为它们的底部完全借助deque,所有操作都由底层的deque供应。()

空间配置器：负责空间的配置与管理。从实现角度看，配置器是一个实现了动态空间配置、空间管理、空间释放的class template。

一般的分配器的std::alloctor都含有两个函数allocate与deallocte,这两个函数分别调用operator new()与delete(),这两个函数的底层又分别是malloc()和free()，但是每次malloc会带来格外开销(每次malloc一个元素都要带有附加信息)

STL优点：1.高可重用性 2.高性能 3.高移植性 4.跨平台 5.数据和操作分离

模板

因为STL使用了大量的模板，所以，在这里对模板进行一些列的讲解

定义

建立一个通用函数或类，其类内部的类型和函数的形式参数类型不具体指定，用一个虚拟的类型来表示。这种通用的方式称为模板。模板是泛型编程的基础，泛型编程即以一种独立于任何特定类型的方式编写代码。

一般使用

函数模板

建立一个通用函数，其函数类型和形参类型不具体指定，用一个虚拟的类型来表示。这个函数称为函数模板。

凡是函数体相同的函数都可以用这个模板来代替，不必定义多个函数，只需在模板中定义一次即可。在调用函数时系统会根据实参的类型来取代模板中的虚拟类型，从而实现了不同函数的功能

**函数模板定义形式:**模板说明+函数定义+函数模板调用

模板说明

template<类型形式参数表>

函数定义

模板说明的类属参数必须在函数定义中出现一次，函数参数表中可以使用类型参数，也可以使用一般形式参数

函数模板调用

  • 类型形式参数表作为返回值时，需注意返回的类型

类模板

跟定义函数模板一样，也可以定义类模板

空间配置器(Allocator)

空间配置器总是隐藏在容器(conatiner)的背后。为什么是空间配置器而不是内存配置器，因为空间不一定是内存，空间也可以是磁盘或其他辅助存储介质。是的，可以写一个allocator直接向硬盘取空间。

主要作用:为容器分配内存空间，容器在实现中需要动态地分配和释放内存，分配器提供了一种统一的接口和策略来管理这些操作。负责跟踪可用内存并提供对分配和释放操作的支持

空间配置器的标准接口

  • allocator::value_type
  • allocator::pointer
  • allocator::const_pointer
  • allocator::reference
  • allocator::const_reference
  • allocator::size_type
  • allocator::difference_type
  • allocator::rebind
  • allocator::allocator() default constructor
  • allocator::allocator(const allocaotr&) copy constructor
  • template < class U >allocator::allocator(const allocator< U > &)泛化的copy constructor
  • allocator::~allocator() default constructor
  • pointer allocator::address(reference x)const:返回某个对象的地址,&x
  • const_pointer allocator::address(const_reference x) const返回某个const对象的地址 a.address(x) =>&x
  • pointer allocator::allocate(size_type n,const void* = 0)配置空间，足以存储n个T对象，第二个参数是个提示，实现上可能会利用它来增进区域性，或忽略
  • void allocator::deallocate(pointer p,size_type n)归还先前配置的空间
  • size_type allocator::max_size()const 返回可成功配置的最大量
  • void allocator::construct(pointer p,const T&x): new (const void*)p T(x)
  • void allocator::destroy(poiner p): p->~T()

具备次配置力(sub-allocation)的SGI空间配置器

一般而言，我们习惯的C++内存配置操作和释放操作

new算式内核2段操作:

  1. 调用::operator new配置内存 alloc::allocate()
  2. 调用Foo::Foo()构造对象内存 alloc::construct()

delete也有两段操作:

  1. 调用Foo::~Foo()将对象析构 alloc::destroy()
  2. 调用::operator delete释放内存 alloc::deallocate()

构造和析构基本工具:construct()和destroy(),对象

construct()接受一个指针p和一个初值value，将初值设定到指针所指的空间上。使用C++的placement new完成。

第二版本的destroy()接受first和last两个迭代器将[first,last)范围内的所有对象析构掉。但范围很大，但每个析构函数都无关痛痒(trivial destructor),调用太多无法的析构函数，对效率是一种损耗。所有先利用value_type()获得迭代器所指对象的类型，利用__type_traits<T>判断。后续讲解__type_traits<>

空间的配置与释放,std::alloc

对象构造前的空间配置和对象析构后的空间释放

如何设计:

  • 向system heap要求空间
  • 考虑多线程(mutil-threads)状态
  • 考虑内存不足时的应变措施
  • 考虑过多”小性区块”可能造成的内存碎片(fragment)问题

考虑小型区块可能造成的内存破碎问题，SGI设计了双层级配置器，第一级直接使用malloc()和free()，第二级配置器则视情况采取不同的选择

SGI为alloc包装了一个接口:

第一级配置器以malloc(),free(),realloc()等C函数执行实际的内存配置、释放、重配置操作，并实现出类似C++ new-handler的机制(内存分配异常的处理机制。并非使用::operator new来配置内存。

C++ new-handler:要求系统再内存配置需求无法被满足时，调用一个你所指定的函数。**::operator new无法完成任务，再丢出std::bad_alloc 异常状态之前，会先调用由客端指定的处理例程。**C++并未提供相应于realloc()内存配置操作

第二级配置器 __default_alloc_template解析

第二级配置器多了一些机制，避免太多小额区块造成内存的碎片。小额区块带来的其实不仅是内存碎片，配置时的额外负担也是大问题。额外区块永远无法避免，毕竟系统要靠这多出来的空间管理内存。但是区块越小，额外负担所占的比例旧越大，越浪费。

第二级配置器的做法:如果区块够大，超过128bytes,就移交第一级配置器处理。当区块小于128bytes，则以内存池(memory pool)管理。

空间配置函数allocate()

身为一个配置器,__default_alloc_template拥有配置器的标准接口函数allocate()。此函数首先判断区块大小，大于128就调用第一级配置器，小于128bytes就检查对应的gree list。如果freelist之内有可用的区块，就直接用，没有就上调8倍数边界，调用refill()
